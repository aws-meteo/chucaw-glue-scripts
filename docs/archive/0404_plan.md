# 🗺️ Plan Detallado de Migración a AWS Glue 5.0 (04-04-2026)

**Objetivo Central:** 
Migrar de manera completa y testeable el build system de *chucaw-glue-scripts* a AWS Glue 5.0. Esto implica utilizar Python 3.11, la imagen oficial Docker basada en Amazon Linux 2023, y el nuevo mecanismo oficial offline (`.gluewheels.zip` + `--additional-python-modules`). 

**Este documento funciona como guía paso por paso (Playbook)** para que cualquier LLM o desarrollador pueda reanudar el trabajo, auditar los cambios realizados o ejecutar las tareas pendientes.

---

## 🏗️ FASE 1: Preparación del Runtime y Configuración [COMPLETADA]

### 1. Actualizar `Dockerfile.glue-builder`
* **Contexto:** Glue 5.0 se ejecuta sobre Amazon Linux 2023. La imagen oficial ya contiene librerías nativas (`curl`, `tar`, `gzip`) y Spark 3.5.4 con Python 3.11.
* **Acción:**
  - Cambiar base image a `public.ecr.aws/glue/aws-glue-libs:5`.
  - Instalar dependencias puras de compilación para `eccodes` (C/C++, CMake) usando `dnf install --allowerasing`.
  - Configurar `USER hadoop` como default.
* **Test de Validación:** `docker build -t glue5-builder:latest .`

### 2. Alinear dependencias base en `pyproject.toml`
* **Contexto:** Glue 5.0 pre-instala versiones específicas (ej. `numpy==2.2.6`, `pandas==2.2.2`, `pyarrow==17.0.0`).
* **Acción:**
  - Actualizar `requires-python = ">=3.11,<3.12"`.
  - Dejar a `xarray`, `cfgrib`, y `eccodes` como dependencias principales.
  - Mover `boto3`, `numpy`, `pandas`, y `pyarrow` a `optional-dependencies[glue-base]` sin anclajes estrictos (`>=X`) para que pip no intente forzar reinstalaciones.

### 3. Limpieza del repositorio
* **Contexto:** Había contradicciones sobre el mecanismo de build y deploy entre la documentación y scripts.
* **Acción:**
  - Archivar scripts de compilación .sh antiguos y docs que mencionan Glue 4.0 al directorio `docs/archive/`.

---

## ⚙️ FASE 2: Build Pipeline (Containerizado) [COMPLETADA]

### 4. Construcción Unificada de Wheels (`build_glue_libs.sh`)
* **Contexto:** Necesitamos un solo script que corra dentro de la imagen Glue 5.0 aislando dependencias y descargando binarios de Linux para Python 3.11.
* **Acciones:**
  - Usar un archivo de *baseline* plano (`lista_de_glue50.txt`) cruzado mediante `grep` para evitar descargar/empaquetar lo que AWS ya provee.
  - Generar el wheel del proyecto `chucaw_preprocessor-*.whl`.
  - Descargar los `manylinux_2_28_x86_64` o `any` permitidos usando `--python-version 311`.
  - Crear el artefacto `glue-dependencies.gluewheels.zip` (comprimiendo la carpeta `wheels/`).
  - Correr una prueba atómica offline nativa instalando el ZIP y referenciando módulos dentro del propio contenedor para validar C-Dependencies (`import eccodes`, etc.).

## 🚀 FASE 3: Despliegue y Ejecución (Pendiente de Ejecución)

*Toda la estructura previa funciona. Ahora las definiciones del Job y del CD deben reflejar este estado.*

### 5. Actualizar Script de Despliegue E2E (`deploy-and-test-glue.ps1`)
* **Problema Actual:** Sube `glue-dependencies.zip` de Glue 4.0 y ejecuta bajo `GlueVersion = "4.0"`.
* **Acción Requerida:**
  - Modificar localizaciones a buscar: `build/glue-dependencies.gluewheels.zip` y `build/chucaw_preprocessor-*.whl`.
  - Cambiar los parámetros del AWS CLI a configurar el Job en Glue 5.0:
    - `--glue-version "5.0"`
    - Reemplazar `--extra-py-files` por `--additional-python-modules s3://.../glue-dependencies.gluewheels.zip,s3://.../chucaw_preprocessor-0.1.0-py3-none-any.whl` (ambos separados por coma).
    - Insertar un param nuevo en el config: `--python-modules-installer-option="--no-index"` (para no intentar resoluciones por red hacia PYPI, evitando timeouts).
  - Testear el flujo ejecutando el script de PS1 para levantar de forma efímera un Job que haga `print("Success")`.

### 6. Actualizar Plantillas en `examples/`
* **Acción Requerida:** 
  - `glue-job-example.json` debe dejar de usar `extra-py-files` y migrar al framework `additional-python-modules`. 
  - Asegurar menciones de la Versión 5.0 en descripciones y nombres.

### 7. Consolidación de `README.md`
* **Acción Requerida:** 
  - Explicar la mecánica clara e implementada de **"zip of wheels"** con `.gluewheels.zip` apoyado en el `public.ecr.aws/glue/aws-glue-libs:5`.
  - Explicar explícitamente y con comandos (como los ya escritos en `boulder.json` y `KANBAN.md`) cómo levantar el entorno la primera vez (Docker build) y cómo correr el empaquetado final (`docker run`).

### 8. Testing Atómico Extra (Smoke Test en Data Real)
* **Objetivo:** Más allá de probar que los imports funcionen, requerimos estar seguros de que el sistema C procese GRIB y convierta a Parquet sin crashear.
* **Acción Requerida:**
  - Dentro de un container Glue 5.0 docker, mapear un archivo GRIB.
  - Ejecutar `/workspace/scripts/glue_jobs/grib_to_platinum_parquet.py` (o similar) y comprobar que el set de datos Parquet se escribe correctamente.

## 🧰 Caja de Herramientas (Comandos de Verificación para el Agente)

Para explorar subsecuentemente o verificar el estado final sin "romper" o si lo olvidas, usa:

1. **Docker check:** `docker images glue5-builder:latest`
2. **Wheels generados:** `ls -la build/` o PowerShell equivalente (Ver que existe `.gluewheels.zip` de >70MB).
3. **Chequeo Offline:**
   ```bash
   docker run --rm -v "${PWD}:/workspace" glue5-builder:latest "/bin/bash" "-c" "pip3.11 install --no-index --find-links /workspace/build/wheels /workspace/build/chucaw_preprocessor-0.1.0-py3-none-any.whl && python3.11 /workspace/verify_glue_deps.py"
   ```
   *(Este test debe retornar `Exit code 0` confirmando que NO se necesita internet para deployar tus librerías en Glue 5.0).*
